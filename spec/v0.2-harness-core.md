# agnoclaw v0.2 Specification (Agno-Aligned Harness Core)

Status: Draft  
Audience: Harness maintainers, platform integrators, enterprise platform team  
Last Updated: 2026-02-24

## 1. Intent

`agnoclaw` v0.2 defines a stable harness core built on Agno SDK primitives.

This repo is not an enterprise control plane. It is the execution harness that can be embedded into one.

Primary goals:
- Keep a local-first, hackable harness for individual users.
- Align runtime semantics with Agno's documented run, hook, session, and event model.
- Expose clear integration seams so a separate enterprise platform repo can add governance, identity, and org workflows.

## 2. Research Basis (Primary Sources)

This spec is based on official Agno documentation and references:
- Running agents (`run`, `arun`, streaming, run events)
- Run context and hooks (pre/post/background hooks)
- Session storage and history management
- Session summaries and context compression
- Skills loading and prompt integration
- AgentOS documentation (overview, interfaces, auth, RBAC, JWT middleware)
- Tracing setup and telemetry guidance
- Agno v2 migration guidance

Reference links are listed in Section 18.

## 3. Scope and Non-Goals

### 3.1 In Scope (v0.2)

- A stable harness API that wraps Agno agent execution.
- Deterministic prompt assembly and skill injection.
- Session and context propagation (`user_id`, `session_id`, metadata).
- Hook points aligned to Agno pre/post hooks.
- Event surface aligned to Agno run events.
- Extensibility contracts for tools, skills, memory, and storage adapters.
- Optional compatibility profile for AgentOS integration.

### 3.2 Out of Scope (v0.2)

- Enterprise admin UI, provisioning, billing, and tenant lifecycle.
- Centralized org policy authoring UX.
- Marketplace backend and skill monetization.
- Cross-tenant orchestration service.

## 4. Product Boundary Decision

### 4.1 What this repo is

- Harness runtime and CLI.
- Developer-facing extension surface.
- Local and embedded execution entrypoint.

### 4.2 What the enterprise platform repo is

- Identity federation and org RBAC administration.
- Fleet operations, approvals, compliance workflows.
- Multi-tenant governance and policy distribution.

### 4.3 AgentOS Position

AgentOS is **not required** for the harness core to be valid.

AgentOS is **recommended as an optional compatibility profile** when you need:
- Hosted/API exposure of agents.
- Built-in auth middleware and role propagation.
- Agent interfaces (including A2A/OpenAI-compatible surfaces).
- Platform-grade operational controls.

Decision: v0.2 treats AgentOS as a profile, not a hard dependency.

### 4.4 When to Adopt AgentOS

Adopt AgentOS if at least two of these become hard requirements:
- API-hosted agent endpoints with centralized auth and permissions.
- org/team/tenant claim propagation from JWT into runtime policy.
- standardized agent interfaces (A2A/OpenAI compatibility surface).
- platform SRE needs for centralized ops, telemetry, and governance.

If these are not required yet, keep the harness core standalone and integrate AgentOS later via Profile C.

## 5. Agno Compatibility Baseline

v0.2 targets Agno's documented v2 API surface.

### 5.1 Required API Alignment

The harness MUST align with these documented Agno capabilities:
- `Agent.run(...)` and `Agent.arun(...)`
- streaming content and intermediate steps
- event streaming via `stream_events=True`
- session/user propagation by passing `user_id` and `session_id`
- hook lifecycle via pre-hooks and post-hooks

### 5.2 Compatibility Rules

- Do not depend on undocumented/private Agno internals.
- If Agno introduces breaking major-version changes, v0.2 compatibility is frozen to the supported major and explicitly version-gated.
- Optional Agno features may be enabled by capability detection and must degrade safely if unavailable.

### 5.3 Design Decisions Anchored to Agno Docs

| Area | Agno documented behavior | v0.2 decision |
|---|---|---|
| Run API | `run/arun` with streaming and `stream_events` | Keep harness run surface aligned; no custom divergent execution API in core |
| Hooks | Pre/post hooks are core; background hooks are separate capability | Treat pre/post as required, background hooks as optional (Profile C) |
| Context | `user_id`, `session_id`, metadata are first-class run/session context | Make context propagation mandatory in embedded mode |
| Events | Run/tool/reasoning events available in event stream | Use Agno event stream as primary telemetry source |
| Sessions | Session storage + history controls are documented patterns | Preserve history controls and avoid ad hoc memory-only behavior |
| Memory pressure | Session summaries and context compression are documented | Include explicit context pressure controls in core config |
| Skills | Skills can inject into system prompt and extend behavior | Keep one-shot skill injection deterministic and reversible |
| Security | AgentOS provides auth/RBAC/JWT middleware patterns | Keep auth/RBAC in optional AgentOS profile, not core harness dependency |

## 6. Runtime Profiles

v0.2 defines three runtime profiles.

### 6.1 Profile A: Core Harness (Required)

Use case: individual users, local CLI, library embedding without AgentOS.

Requirements:
- Works with local workspace files and local skill folders.
- Uses Agno agent runtime directly.
- Supports per-run skill prompt injection without persistent prompt corruption.
- Preserves session semantics (including session reset/rotation).

### 6.2 Profile B: Embedded Platform (Required)

Use case: another service embeds harness runtime.

Requirements:
- Accept external context (`user_id`, `session_id`, metadata, optional org fields).
- Emit structured run/tool events to platform sinks.
- Allow policy interception at run and tool boundaries.
- Keep storage/memory backend pluggable.

### 6.3 Profile C: AgentOS Compatibility (Optional but Strategic)

Use case: enterprise deployment path.

Requirements:
- Map incoming auth claims to harness execution context.
- Respect AgentOS auth/RBAC decisions when present.
- Keep hooks and event surfaces compatible with AgentOS operational tooling.
- Avoid introducing APIs that block future AgentOS-hosted operation.

## 7. Core Runtime Contract

### 7.1 Canonical Run Request

Harness run API MUST support:
- prompt/input content
- `stream` toggle
- `stream_events` toggle
- optional `user_id`
- optional `session_id`
- optional metadata/tags
- optional one-shot skill activation

### 7.2 Canonical Run Result

- Non-streaming: returns full assistant result.
- Streaming: yields content and (optionally) run events.
- Errors: surfaced as typed harness errors with stable codes.

### 7.3 Session Behavior

- If `session_id` is omitted, harness may create one.
- A clear/reset action MUST rotate to a new session ID and drop old context from active run state.
- History retrieval MUST reference the effective active session ID.

## 8. Execution Context Contract

v0.2 context envelope:

```python
@dataclass(frozen=True)
class ExecutionContext:
    user_id: str | None
    session_id: str | None
    workspace_id: str | None
    tenant_id: str | None = None
    org_id: str | None = None
    team_id: str | None = None
    roles: tuple[str, ...] = ()
    scopes: tuple[str, ...] = ()
    request_id: str | None = None
    trace_id: str | None = None
    metadata: dict[str, Any] = field(default_factory=dict)
```

Rules:
- Immutable per run turn.
- Child operations may append run-local metadata but may not mutate identity fields.
- When AgentOS claims are available, they populate context fields; otherwise fields remain optional in local mode.

## 9. Hook and Lifecycle Semantics

### 9.1 Lifecycle (Single Run)

1. Build effective system prompt (workspace + identity + policy + optional skill prompt).
2. Invoke pre-run hook chain.
3. Execute Agno run/arun.
4. Emit/forward run events while executing.
5. Invoke post-run hook chain.
6. Persist session/history/memory outputs.

### 9.2 Hook Types

- Pre-hooks: inspect/transform input and enforce policy before model invocation.
- Post-hooks: inspect/transform output, add follow-up actions, audit logging.
- Background hooks: optional; considered AgentOS profile functionality, not core requirement.

### 9.3 Hook Contracts

Hook failures must be categorized:
- `HOOK_POLICY_DENY`
- `HOOK_TRANSFORM_ERROR`
- `HOOK_INTERNAL_ERROR`

Policy-deny behavior is configurable:
- fail-closed (block run)
- fail-open with warning (local/dev profile only)

## 10. Event Model (Agno-Aligned)

### 10.1 Event Sources

v0.2 events come from:
- Agno run event stream (`stream_events=True`)
- Harness lifecycle events (prompt built, skill loaded, policy decision)

### 10.2 Required Event Coverage

At minimum, v0.2 must represent:
- run started
- response content chunks (for streaming runs)
- tool call started/completed
- reasoning step events (if model/runtime emits)
- run completed/failed
- policy decisions (harness layer)
- skill activation/load decisions

### 10.3 Delivery Guarantees

- At-least-once delivery to sinks.
- Best-effort default sink mode.
- Optional fail-closed mode for regulated environments.
- Deduplication key: `event_id`.
- Ordering guaranteed per run, not globally.

## 11. Prompt Assembly and Skill Injection

### 11.1 Base Prompt Sources

Prompt builder may compose from workspace files (for example: `AGENTS.md`, `SOUL.md`, `IDENTITY.md`, `MEMORY.md`, `TOOLS.md`) with configured caps.

### 11.2 Safety and Determinism

- Prompt caps MUST be enforced (per-file and total limits).
- Memory bootstrap lines MUST be capped.
- Skill injection for a run must be temporary and restored after the run.
- Plan mode and learning toggles must be explicit prompt options, not accidental state leaks.

### 11.3 Skill Precedence

1. one-shot invocation skill (current run only)
2. workspace skill override
3. bundled skill
4. installed user skill

## 12. Tools and Side-Effect Boundaries

### 12.1 Tool Execution

- Tool interfaces remain async-first.
- Tool calls are auditable and evented.
- Tool errors map to stable error families.

### 12.2 Guardrails

- File/path boundaries must be enforceable.
- Shell timeout limits are mandatory.
- Networked tools are policy-gated.
- Sensitive output redaction can be enforced in post-tool policy.

### 12.3 AgentOS Consideration

If AgentOS policy is present, harness-side policy must not bypass it. Harness policy can only further restrict.

## 13. Session, History, and Memory Strategy

### 13.1 Session Storage

v0.2 should support:
- local persistent session history for standalone mode
- pluggable DB-backed storage for embedded/platform mode

### 13.2 History Controls (Agno-Aligned)

Expose and preserve these behavior patterns:
- include recent history in context (`add_history_to_context` pattern)
- bound history inclusion (`num_history_runs` pattern)
- explicit read-history tooling for agent self-retrieval

### 13.3 Context Pressure Controls

v0.2 should include policy/config for:
- session summary usage when history is long
- context compression of large tool outputs
- hard token/size caps before model call

## 14. Security and Identity

### 14.1 Core Harness Security Baseline

- Least-privilege tool defaults.
- Secrets must not be written to persistent logs/events by default.
- Context identities are explicit and traceable.

### 14.2 AgentOS Security Profile

When AgentOS is used:
- JWT middleware claims should map into execution context.
- Team/org/tenant roles should be available for policy checks.
- Endpoint-level permissions and route protection remain platform-owned.

### 14.3 Multi-Tenant Isolation Rule

Cross-tenant and cross-user context access is denied by default. Any override must be explicit policy and auditable.

## 15. Observability and Tracing

### 15.1 Required Observability

- Structured run events.
- Structured tool execution logs.
- Error code + category + retryability.

### 15.2 Tracing

Harness should support Agno tracing setup for production integrations and keep trace IDs in context/events where available.

### 15.3 Minimal Telemetry Contract

Each run event should include:
- `event_id`
- `event_type`
- `occurred_at`
- `run_id`
- `session_id` (if available)
- `user_id` (if available)
- `trace_id` (if available)
- payload

## 16. Testing and Acceptance Criteria

### 16.1 Contract Tests (Required)

- run sync/async parity
- streaming and stream-events behavior
- hook ordering and failure handling
- session clear/reset rotates session ID
- prompt restoration after skill injection
- context caps enforcement
- config deep merge behavior for nested sections
- event schema validation and dedupe key presence

### 16.2 Profile-Specific Tests

Profile A (Core):
- local workspace bootstrap
- bundled skills discovery/packaging

Profile B (Embedded):
- external context propagation
- pluggable storage/policy adapters

Profile C (AgentOS optional):
- auth claim to context mapping
- RBAC-driven tool denial path
- interface compatibility smoke tests (A2A/OpenAI endpoint exposure as applicable)

### 16.3 Completion Gate for v0.2

v0.2 is complete when:
- Required contract tests pass.
- Required profile A/B behavior is implemented and documented.
- AgentOS compatibility profile has a documented adapter plan and at least one integration smoke test.

## 17. Delivery Plan

### Phase 1: Core Alignment

- Normalize run API around Agno run/arun semantics.
- Finalize deterministic prompt and skill behavior.
- Ship typed errors and stable event envelope.

### Phase 2: Embedding Surfaces

- Stabilize policy, storage, and event sink adapters.
- Publish embedding docs and migration notes.

### Phase 3: AgentOS Compatibility

- Add optional auth-context adapter.
- Add compatibility examples for API-hosted deployment.
- Validate RBAC + event observability flow under AgentOS.
