# agnoclaw v0.8 Direction: Harness SDK, Optional Server, Packs

Status: Direction draft
Audience: agnoclaw maintainers and product integrators
Last Updated: 2026-05-25

## 1. Thesis

`agnoclaw` remains an embeddable Python harness SDK first.

AgentOS, remote runtimes, package systems, and server APIs adapt to `AgentHarness`;
they do not replace it.

The v0.8 direction is:

```text
agnoclaw-core      = embeddable harness SDK
agnoclaw-agentos   = optional AgentOS runtime adapter
agnoclaw-server    = thin convenience layer over AgentOS-compatible runtime export
agnoclaw-packs     = Python-native packaging for tools, skills, hooks, policies, and context
```

This keeps agnoclaw useful as a personal SDK across products while allowing it to
run as a service when the product needs hosted sessions, streaming, approvals,
scheduler, observability, or external clients.

## 2. Non-Negotiable Boundary

`AgentHarness` is the canonical runtime.

All other surfaces must preserve its behavior:

- system prompt and workspace context
- skills and skill enforcement
- policy checkpoints
- permission modes
- runtime guardrails
- event emission
- memory/session behavior
- tool and backend routing

No server path may call the underlying Agno `Agent` directly in a way that bypasses
agnoclaw runtime controls.

## 3. Why This Exists Now

The agent-harness market has moved from "agent loop library" to "runtime substrate."

Recent signals:

- Agno AgentOS now provides production server concerns: sessions, SSE-compatible
  streaming, traces, JWT/RBAC, approvals, scheduler, MCP exposure, and multi-framework
  adapters.
- Agno context providers now provide a first-party shape for external context sources
  with narrow `query_<id>` / `update_<id>` tool surfaces.
- Cursor SDK shows that developer adoption is strongly helped by a small programmatic
  API around local/cloud agent execution and streaming events.
- Pi's TypeScript extension/package ecosystem shows the value of composable userland
  packs that bundle tools, skills, lifecycle hooks, UI, commands, and prompt templates.
- OpenAI Agents SDK and LangGraph reinforce durable execution, sandbox separation,
  traceability, and resumable workflow state as mainstream harness capabilities.

agnoclaw should absorb the useful patterns without becoming a hosted platform.

## 4. Product Shape

### 4.1 Embedded first

The simplest use case remains:

```python
from agnoclaw import AgentHarness

harness = AgentHarness(...)
result = await harness.arun("Analyze this deal memo")
```

This path must not require AgentOS, FastAPI, a server process, external scheduler,
or network-accessible runtime.

### 4.2 Server-capable

Products that need a server should be able to opt in:

```python
from agnoclaw import AgentHarness
from agnoclaw.runtime.agentos import create_agentos_app

harness = AgentHarness(name="deal-agent", ...)
app = create_agentos_app([harness], scheduler=True, approvals=True)
```

This is a convenience adapter around AgentOS-compatible runtime behavior. It is not
a separate agnoclaw control plane.

### 4.3 Remote-callable

Remote clients should use an agnoclaw-shaped API:

```python
from agnoclaw.remote import RemoteHarnessClient

client = RemoteHarnessClient("https://agents.example.com")
run = await client.arun("Prepare the weekly risk summary", stream=True)
```

The remote surface should preserve the same concepts: context, metadata, stream events,
skills, permission states, and typed errors.

## 5. Workstream A: Agno Context Provider Bridge

### Goal

Let users pass Agno context providers directly into `AgentHarness`.

### Proposed API

```python
harness = AgentHarness(
    context_providers=[workspace_provider, slack_provider, db_provider],
    dependencies={"tenant_config": {...}},
    add_dependencies_to_context=False,
)
```

### Responsibilities

- Add provider tools to the harness tool list.
- Add provider instructions to the system prompt in a bounded section.
- Support provider lifecycle setup/close where needed.
- Propagate agnoclaw `ExecutionContext` into Agno run metadata/dependencies.
- Preserve agnoclaw policy, permission, event, and guardrail boundaries.
- Avoid duplicate tool names and surface a clear error or warning.

### Non-goals

- Replace current agnoclaw tools with Agno context providers by default.
- Force users onto Agno's provider model.
- Remove existing workspace context files or skill injection.

## 6. Workstream B: AgentOS Runtime Export

### Goal

Make agnoclaw server-capable through AgentOS while preserving `AgentHarness` as the
runtime boundary.

### Proposed API

```python
agent = harness.as_agentos_agent()
app = create_agentos_app([harness], enable_mcp_server=True)
```

### Required behavior

- Calls route through `harness.run()` / `harness.arun()`.
- Stream events map to AgentOS-compatible SSE events.
- AgentOS request/JWT claims map into `ExecutionContext`.
- AgentOS approvals can satisfy agnoclaw permission requests.
- AgentOS traces include agnoclaw run IDs and event metadata.
- AgentOS scheduler can invoke harness runs without custom glue code.

### Optional behavior

- `RemoteHarnessClient` can call AgentOS endpoints with agnoclaw-shaped helpers.
- AgentOS MCP server can expose harness agents as MCP tools.
- Multiple harnesses can be registered under stable agent IDs.

### Non-goals

- Build a custom agnoclaw control plane.
- Reimplement AgentOS session, trace, scheduler, approval, or MCP endpoints.
- Make AgentOS a hard dependency of `agnoclaw-core`.

### Endpoint audit

AgentOS already owns the primary product API surface:

- run/list/get for agents, teams, and workflows
- session CRUD
- memory CRUD
- knowledge add/search/list/get/delete
- eval CRUD
- metrics and traces
- schedule CRUD, enable/disable, trigger, and run history
- approval list/get/status/count/resolve/delete
- MCP server exposure via `/mcp`

agnoclaw should not add competing versions of those endpoints. Server export should
register the harness through AgentOS so normal execution uses:

```text
POST /agents/{agent_id}/runs
POST /agents/{agent_id}/runs/{run_id}/cancel
GET  /sessions/...
GET  /traces/...
GET  /approvals/...
GET  /schedules/...
```

agnoclaw may add a small optional admin/debug router for concepts it owns above Agno:

```text
GET    /agnoclaw/harnesses
GET    /agnoclaw/harnesses/{harness_id}/capabilities
GET    /agnoclaw/harnesses/{harness_id}/runtime
GET    /agnoclaw/harnesses/{harness_id}/events/{run_id}

GET    /agnoclaw/harnesses/{harness_id}/sandboxes/{session_id}
GET    /agnoclaw/harnesses/{harness_id}/sandboxes/{session_id}/files
GET    /agnoclaw/harnesses/{harness_id}/sandboxes/{session_id}/artifacts/{artifact_id}
POST   /agnoclaw/harnesses/{harness_id}/sandboxes/{session_id}/snapshot
POST   /agnoclaw/harnesses/{harness_id}/sandboxes/{session_id}/reset

GET    /agnoclaw/harnesses/{harness_id}/skills
GET    /agnoclaw/harnesses/{harness_id}/packs
GET    /agnoclaw/harnesses/{harness_id}/policies
GET    /agnoclaw/harnesses/{harness_id}/permissions
```

Rules for these endpoints:

- They are optional and disabled by default unless `include_agnoclaw_admin=True`.
- They mount under `/agnoclaw` to avoid conflict with AgentOS API routes.
- They require admin/debug scopes when AgentOS RBAC is enabled.
- They expose observability and lifecycle controls, not a second agent-run API.
- Sandbox write/reset actions must route through agnoclaw policy and event emission.
- Artifact/file endpoints should default to read-only listing/download, not arbitrary
  filesystem mutation.
- The same capabilities should also be available in embedded mode as Python methods,
  so server mode is never the only control surface.

## 7. Workstream C: Pack System v1

### Goal

Create a Python-native answer to Pi packages: a trusted, inspectable bundle of
agent harness capabilities.

### Pack contents

A pack may provide:

- tools
- skills
- context providers
- lifecycle hooks
- policy engines or policy rules
- prompt templates
- slash commands / CLI commands
- runtime backends
- UI adapters
- examples and tests

### Manifest shape

```toml
# agnoclaw-pack.toml
name = "my-product-agent-pack"
version = "0.1.0"
description = "Deal workflow tools and skills"

[provides]
skills = ["skills/"]
tools = ["my_pack.tools:register"]
hooks = ["my_pack.hooks:register"]
context_providers = ["my_pack.context:register"]
policies = ["my_pack.policies:register"]
commands = ["my_pack.commands:register"]

[trust]
default = "local"
requires_code_execution = true
```

### CLI shape

```bash
agnoclaw pack list
agnoclaw pack inspect ./agent-pack
agnoclaw pack install git+https://github.com/acme/agent-pack
agnoclaw pack trust my-product-agent-pack
agnoclaw pack remove my-product-agent-pack
```

### Runtime behavior

- Packs are disabled by default unless explicitly installed or referenced.
- Packs have trust levels matching existing skill/plugin trust concepts.
- Code-executing packs require explicit trust.
- Pack-provided hooks and policies emit events when they run.
- Pack load failures are visible and diagnosable.

### Non-goals

- Build a public marketplace in v1.
- Support arbitrary npm package loading in Python runtime.
- Allow untrusted code execution silently.

## 8. Workstream D: SDK Ergonomics

### Goal

Make the everyday API feel like a modern harness SDK without breaking the current API.

### Additive API ideas

```python
harness = await AgentHarness.create(...)
session = harness.session(user_id="u1", workspace_id="repo-a")
run = await session.send("Fix the failing test", stream=True)

async for event in run.events():
    ...
```

This should be sugar over existing `run/arun`, not a separate execution engine.

### Event contract

Expose a stable event stream for:

- run lifecycle
- prompt built
- model request lifecycle
- tool lifecycle
- permission requested/resolved
- approval pause/continue
- context provider query/update
- skill load/fork/dispatch
- compaction/session summary
- scheduler invocation

## 9. Debt To Clear While Doing This

From current repo state:

- Update harness gap analysis for Agno v2.6.x, AgentOS scheduler, AgentOS MCP server,
  context providers, HITL approvals, and multi-framework adapters.
- Elevated execution maps to an explicit harness SDK command contract with
  approval + policy + audit events, a single-command CLI `/elevated <cmd>`
  directive, and OpenClaw-style session-wide `/elevated on|ask|full|off`
  defaults for subsequent bash tool calls.
- Add broader hook-pack lifecycle coverage instead of only run/tool hooks.
- Formalize persistent scheduler backend abstraction.
- Persist local background task metadata so embedded bash task output/status can
  be recovered across executor instances.
- Keep sandbox/provider abstraction coherent with local, container, and remote runtimes.
- Structured plan/user-question tooling is now an embeddable primitive through
  `AskUserQuestion`, `ExitPlanMode`, and harness plan signal methods.

## 10. Suggested Implementation Order

1. Context provider bridge.
2. AgentOS export.
3. Pack system v1.
4. SDK ergonomics layer.
5. Remote client.
6. Persistent scheduler backend.

This order keeps the blast radius small. Each step preserves embedded usage and adds
one optional capability surface.

## 11. Compatibility Rules

- Existing `AgentHarness(...)` constructor behavior remains valid.
- Existing `run()` / `arun()` behavior remains valid.
- Existing built-in tools remain available.
- Existing plugin entry points continue to work.
- Existing skills and ClawHub integration continue to work.
- AgentOS and packs are optional extras.
- Any breaking change requires a migration note and deprecation window.

## 12. Decision Record

Decision: agnoclaw becomes server-capable, not server-first.

Rationale:

- Server mode is valuable for products, teams, schedulers, approvals, observability,
  and external clients.
- AgentOS already owns much of the production runtime surface.
- agnoclaw's durable value is a Python-first harness contract the maintainer can carry
  across products without depending on one vendor SDK or editor runtime.

Implication:

- Build adapters and pack surfaces around `AgentHarness`.
- Avoid creating a competing platform layer unless AgentOS becomes insufficient for a
  specific, concrete agnoclaw use case.
